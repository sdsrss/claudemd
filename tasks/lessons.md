- 2026-05-09 [plugin-update-needs-marketplace-refresh-first]: Recommended user run `/plugin uninstall claudemd@claudemd && /plugin install claudemd@claudemd && /reload-plugins` to upgrade from v0.6.0 → v0.6.3. After all three commands ran successfully, `claudemd-status` still reported `version: 0.6.0` — uninstall + install only repopulates from `~/.claude/plugins/marketplaces/claudemd/.claude-plugin/marketplace.json` (a local cache), which itself was still pinned at v0.6.0. CC does not auto-refresh marketplace caches on install. Memory `reference_plugin_update_manual_refresh.md` already documented the correct four-step sequence (marketplace update → uninstall → install → reload), but I dropped the marketplace-update step when reproducing it from memory. → **Rule**: when recommending plugin upgrade commands, always include `/plugin marketplace update <name>` as step 1 before uninstall+install. The local marketplace cache is the source of truth for `/plugin install` version resolution; without refreshing it, install is a no-op masquerading as an upgrade. Verify post-upgrade by checking `~/.claude/plugins/cache/<plugin>/<plugin>/<expected-version>/` exists, not just trusting the install command's success message.

- 2026-04-29 [diagnostic-step-bsd-vs-gnu-stat]: v0.5.0 §1.A diagnostic step used `stat -f %m` (BSD form) to print ref/marker mtimes on macOS, but the CI install step prepends `$(brew --prefix coreutils)/libexec/gnubin` to PATH, so `stat` resolves to GNU `stat` which rejects `-f`. CI run 25075330298 macOS log: `stat: cannot read file system information for '%m': No such file or directory`. Mtime delta line printed `0s` because arithmetic on the error string fell through to 0. → **Rule**: any portable diagnostic on a CI matrix where macOS uses brew gnubin must call `stat -c %Y` (GNU) — NOT `stat -f %m` (BSD). When the script needs to support both real-BSD-stat AND gnubin-on-macOS, use the existing `hooks/lib/platform.sh::platform_stat_mtime` wrapper or branch on `command -v gstat`. Memory `feedback_macos_shell_portability.md` already covered the inverse case (BSD on default-PATH macOS) — this entry adds the "gnubin-replaces-BSD" case.

- 2026-04-29 [macos-tmp-essentially-empty]: v0.5.0 §1.A diagnostic step on macos-15-arm64 GH runner reported `find /tmp -maxdepth 1 | wc -l` = 1 (only `/tmp` itself, no children). This refutes hypothesis C from the v0.4.1/v0.4.2 root-cause candidates ("`head -n 50` truncation under macOS /tmp churn") — there is no churn to truncate. The runner /tmp is essentially virgin at test start. Hypotheses still in play: A (mkdir silent fail), B (mtime sub-second / cross-FS comparison), E (bash 3.2 vs 5.x subshell). The diagnostic also confirmed `BASH_VERSION=3.2.57(1)-release` (system /bin/bash, NOT homebrew bash 5.x) and `/tmp realpath: /private/tmp`. → **Rule**: when ranking candidate root causes for CI-only macOS failures, the runner image's actual state matters more than locally-reasoned heuristics. A diagnostic CI run captures one piece of ground truth per execution; don't speculate about churn / SIP / sandbox without a probe.

- 2026-04-29 [macos-ci-tmp-flake]: v0.4.1 added sandbox-disposal.test.sh Case 8 (`/tmp/claudemd-* still flagged`) that PASSED on Linux but FAILED reproducibly on GitHub Actions macOS runners with stderr empty (FOUND list empty in hook). v0.4.2 patched with `touch NOW + sleep 1 + mkdir` (mtime edge defense) + basename-grep (symlink path-form defense) — no change in outcome. Without macOS real-machine access, root cause unconfirmed: candidates remaining are GH runner /tmp write-permission silent-fail, BSD vs GNU find divergence under brew gnubin PATH, runner /tmp churn racing the hook, or sandbox/SIP behavior unique to hosted runners. → **Rule**: macOS-specific filesystem tests MUST be reproduced on a real macOS machine before landing; CI-only failures get conditional skip + lessons entry rather than CI-iterated patch-spam (3-strike rule, §EXT §6). Test design that can be unit-mocked (no real /tmp dependency) is preferred over real-FS cross-platform assertions.
